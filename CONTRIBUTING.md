# 贡献指南

感谢参与 SCOPE Group Web 的维护。本项目同时包含公开双语网站、GitHub Pages 自动
发布流程和备用 Boda 部署工具；提交 Issue 或 Pull Request 前，请先确认改动属于网站
内容、Hugo 实现、Python 工具还是生产运营，并遵守相应边界。

## 开始之前

1. 阅读 [`README.md`](README.md)，在本地运行项目。
2. 阅读 [`AGENTS.md`](AGENTS.md) 了解仓库规则和安全边界。
3. 修改页面、样式或公开文案前阅读 [`DESIGN.md`](DESIGN.md)。
4. 根据任务选择 [`docs/README.md`](docs/README.md) 中的开发、内容或运营指南。
5. 搜索现有 Issue 和 Pull Request，避免重复工作。

不得在 Issue、PR、提交、日志或截图中包含密码、OTP 种子、会话 Cookie、`.env`
内容、私人材料或其他凭据。发现安全问题时，不要公开披露可利用细节；请通过受信任的
私下渠道联系仓库维护者。

## 提交 Issue

Issue 应描述一个可以判断和处理的问题。将多个无关问题拆成独立 Issue；纯咨询也应
说明背景和期望获得的决定。

### Issue 必要信息

- 清晰、可搜索的标题。
- 问题类型：缺陷、内容更正、功能建议、文档或运营问题。
- 受影响的公开 URL、仓库路径或命令。
- 当前行为、期望行为和影响。
- 可复现步骤；无法稳定复现时说明发生频率和已排除条件。
- 适用的环境信息，例如操作系统、浏览器、视口、Hugo 或 Python 版本。
- 内容问题需附可信来源；不要根据推测补写科学主张、履历或日期。
- 截图应裁去账号、Cookie、Token、后台站点数据和其他敏感信息。

### Issue checklist

提交前请逐项确认，并在 Issue 中保留这份清单：

- [ ] 我已搜索现有 Issue 和 Pull Request，没有发现相同问题。
- [ ] 我提供了受影响的 URL、文件路径或命令。
- [ ] 我写明了当前行为、期望行为和实际影响。
- [ ] 我提供了最小复现步骤，或说明了为何无法稳定复现。
- [ ] 我标明了受影响的语言：英文、中文或两者。
- [ ] 如果涉及公开事实或科学内容，我提供了可核验来源。
- [ ] 如果涉及页面显示，我提供了浏览器、视口和必要截图。
- [ ] 我确认正文、日志和附件中不含凭据、会话信息或私人材料。
- [ ] 该 Issue 不要求未经授权的 Boda 写操作或生产探测。

无法满足某项时，不要直接删除；请写明“不适用”或缺失原因。

## 开发流程

从当前目标分支创建一个聚焦分支，保持每次改动范围单一。不要顺手重构无关模块，
也不要修改可选的只读旧站快照 `source/xulm.pku.edu.cn/`。

### 网站改动

- Hugo 命令默认从仓库根目录执行，并使用 `--source site`。
- 中英文对应页面保持相同的相对路径和 `translationKey`。
- 两种语言中的事实、主张、限定语、链接、人物和日期必须等价，但不要求逐句直译。
- 保留现有公开 URL；需要改变 URL 时，在 PR 中明确记录兼容和发布影响。
- 图片必须具有准确的 alt 文本；不要添加未经批准的科学图示或宣传性文案。

本地预览和构建：

```sh
hugo server --source site
hugo --source site --minify
```

### Python 和部署工具改动

安装锁定依赖并运行测试：

```sh
uv sync --locked --dev
uv run pytest tests
```

正常生产发布由 `.github/workflows/pages.yml` 在改动进入 `main` 后自动完成。普通 PR
只需完成本地构建和测试，不得手工部署 GitHub Pages。

对 Boda CLI、发布脚本或工作流的改动还应运行与改动相关的本地只读检查。Boda 仅作为
备用或应急入口保留；不得为了验证 PR 而执行远端上传、CRUD smoke、部署或全局发布。
生产写操作需要独立授权，并遵循 [`BODA_DEPLOYMENT.md`](BODA_DEPLOYMENT.md)。

## 提交 Pull Request

PR 标题应简洁描述用户可见结果或工程变化，例如：

- `修正中文研究页面的图注`
- `Add publication highlight metadata`
- `Fail closed on mismatched Boda release state`

PR 描述至少包括：

- **背景**：为什么需要修改，关联哪个 Issue。
- **改动**：具体改变了哪些行为、内容或接口。
- **验证**：实际运行过的命令及结果；不能运行的检查需说明原因。
- **截图**：用户可见改动应提供桌面和移动视图，或说明为何不适用。
- **风险**：URL、双语内容、依赖、部署、数据或兼容性影响。

### PR checklist

提交或请求审查前，请逐项确认：

- [ ] PR 只处理一个聚焦问题，并关联对应 Issue（如有）。
- [ ] 我已阅读并遵守 `AGENTS.md` 和相关开发或运营文档。
- [ ] 我没有修改 `source/xulm.pku.edu.cn/` 或提交生成的 `site/public/`。
- [ ] 中英文内容在事实、主张、限定语、链接、人物和日期上保持等价。
- [ ] 对应双语页面使用相同的相对路径和 `translationKey`。
- [ ] 我没有虚构或强化科学结论、履历、奖项、日期、方法或成果。
- [ ] 用户可见改动符合 `DESIGN.md`，并检查了桌面和移动布局。
- [ ] 新增或修改的图片具有准确 alt 文本，交互控件可使用键盘访问。
- [ ] 我运行了 `hugo --source site --minify`，且构建无错误。
- [ ] Python 或部署工具发生变化时，我运行了 `uv run pytest tests`。
- [ ] 我在 PR 中列出了实际执行的验证命令和结果。
- [ ] 如果改变公开 URL、依赖、配置或数据结构，我已说明迁移和兼容影响。
- [ ] 我没有提交凭据、OTP、Cookie、`.env`、私人材料或无关生成文件。
- [ ] 我没有为验证 PR 手工部署 GitHub Pages，也没有执行未经授权的 Boda 上传、CRUD smoke 或生产部署。
- [ ] 文档、命令或工作流行为发生变化时，我同步更新了相关文档。

不适用的项目应保留并标注 `N/A`，不要通过删除检查项隐藏未执行的验证。

## 审查标准

维护者主要检查：

1. 行为是否正确，是否引入回归或破坏公开 URL。
2. 中英文内容是否等价，科学和个人信息是否有可靠来源。
3. 实现是否符合现有 Hugo、Python 和部署工具约定。
4. 测试和浏览器验证是否覆盖改动风险。
5. 是否遵守生产部署、凭据和只读快照边界。
6. 文档是否足以让下一位维护者理解和复现改动。

CI 通过不代表 PR 自动可合并。公开内容、视觉变化和生产工具仍需要人工审查。
